# 养老院应用架构说明

## 📋 应用概述
养老院管理系统包含家属端和管理端两个子系统，提供养老院日常管理、家属关爱、数据统计等核心功能。

## 🏠 家属端页面

### 页面列表
| 页面ID | 页面名称 | 主要功能 | 数据模型 | 导航来源 |
|--------|---------|---------|---------|---------|
| login | 登录引导 | 引导用户进入微信登录 | 无 | 应用入口 |
| wechat-login | 微信登录 | 微信授权登录、获取手机号 | employee、branding | login 自动跳转 |
| bind-senior | 绑定长者 | 绑定家属与长者关系 | family_members | wechat-login 成功后 |
| home | 家属首页 | 长者概况、最新通知、快捷入口 | elders、daily_reports、leave_requests、bills | bind-senior 成功后 |
| care | 护理日报 | 查看护理日报详情 | daily_reports | TabBar 导航 |
| leave | 请假申请 | 申请外出请假、查看记录 | leave_requests | TabBar 导航 |
| bill | 缴费账单 | 查看账单明细、缴费 | bills、branding | TabBar 导航 |

### 导航流程
```
登录流程：
login → wechat-login → bind-senior → home

日常使用：
home ↔ care ↔ leave ↔ bill (通过 TabBar 底部导航切换)
```

### 权限控制
- **用户角色**：family（家属）
- **权限验证**：所有家属页面都会检查 `props.$w.auth.currentUser.type === 'family'`
- **未授权处理**：重定向到 login 页面并显示权限提示

## 🛠️ 管理端页面

### 页面列表
| 页面ID | 页面名称 | 主要功能 | 数据模型 | 导航来源 |
|--------|---------|---------|---------|---------|
| admin-home | 管理首页 | 数据统计、待办事项、快捷入口 | daily_reports、leave_requests、bills、elders | AdminTabBar 导航 |
| admin-daily | 日报审核 | 审核护理日报、编辑日报 | daily_reports | AdminTabBar 导航 |
| admin-leave | 请假审核 | 审核请假申请 | leave_requests | AdminTabBar 导航 |
| admin-bill | 缴费审核 | 审核账单、生成账单 | bills | AdminTabBar 导航 |
| admin-elder | 老人管理 | 老人信息管理、新增/编辑/删除 | elders | AdminTabBar 导航 |
| admin-data | 数据导入 | 批量导入老人、账单数据 | elders、bills | AdminTabBar 导航 |
| admin-brand | 品牌设置 | 配置养老院名称、Logo、主题色 | branding | AdminTabBar 导航 |

### 导航流程
```
管理端所有页面通过 AdminTabBar 底部导航切换：
admin-home ↔ admin-daily ↔ admin-leave ↔ admin-bill ↔ admin-elder ↔ admin-data ↔ admin-brand
```

### 权限控制
- **用户角色**：nurse（护工）、staff（文员）、admin（管理员）
- **权限验证**：所有管理端页面都会检查 `['nurse', 'staff', 'admin'].includes(user.type)`
- **未授权处理**：重定向到 login 页面并显示权限提示

## 📊 数据模型结构

### 核心数据模型

#### 1. employee（员工/家属表）
- **用途**：存储员工和家属信息
- **关键字段**：
  - `openid` - 微信唯一标识
  - `name` - 姓名/昵称
  - `phone` - 手机号
  - `role` - 角色（家属、护工、文员、管理员）
  - `status` - 在职状态
- **关联**：与家属登录、用户身份验证相关

#### 2. elders（老人信息表）
- **用途**：存储老人基本信息
- **关键字段**：
  - `name` - 姓名
  - `age` - 年龄
  - `roomNumber` - 房间号
  - `careLevel` - 护理等级
  - `healthStatus` - 健康状态
  - `emergencyContact` - 紧急联系人
  - `emergencyPhone` - 紧急电话
  - `status` - 在院状态
- **关联**：家属端首页显示、管理端老人管理

#### 3. daily_reports（护理日报表）
- **用途**：存储每日护理记录
- **关键字段**：
  - `elderId` - 关联老人ID
  - `date` - 日期
  - `meals` - 餐食记录
  - `medications` - 用药记录
  - `activities` - 活动记录
  - `mood` - 心情状态
  - `status` - 审核状态（pending、approved、rejected）
- **关联**：家属端护理日报、管理端日报审核

#### 4. leave_requests（请假申请表）
- **用途**：存储请假申请记录
- **关键字段**：
  - `elderId` - 关联老人ID
  - `reason` - 请假事由
  - `startDate` - 开始日期
  - `endDate` - 结束日期
  - `status` - 审核状态（pending、approved、rejected）
- **关联**：家属端请假申请、管理端请假审核

#### 5. bills（账单表）
- **用途**：存储月度账单信息
- **关键字段**：
  - `elderId` - 关联老人ID
  - `month` - 月份
  - `totalAmount` - 总金额
  - `items` - 账单明细
  - `status` - 缴费状态（unpaid、paid、pending）
  - `dueDate` - 缴费期限
- **关联**：家属端缴费账单、管理端缴费审核

#### 6. branding（品牌配置表）
- **用途**：存储养老院品牌配置信息
- **关键字段**：
  - `name` - 养老院名称
  - `slogan` - 宣传口号
  - `logoUrl` - Logo图片地址
  - `primaryColor` - 主题颜色
  - `contactPhone` - 联系电话
  - `contactAddress` - 联系地址
  - `description` - 详细描述
- **关联**：NursingHomeBrand 组件、品牌设置页面

### 数据模型命名规范

#### ✅ 标准命名（推荐使用）
- `elders` - 老人信息（复数形式，表示集合）
- `daily_reports` - 护理日报（下划线分隔）
- `leave_requests` - 请假申请（下划线分隔）
- `bills` - 账单（复数形式）
- `branding` - 品牌配置（单数形式，表示单一配置）
- `employee` - 员工（单数形式，表示员工类别）

#### ⚠️ 重复命名（需要统一）
- `elder` / `elders` - 老人信息存在两个集合
- `daily_report` / `daily_reports` - 日报存在两个集合
- `leave` / `leave_requests` - 请假存在两个集合
- `bill` / `bills` - 账单存在两个集合

#### 📝 未使用集合
- `daily` - 空集合，未使用
- `daily_report` - 空集合，未使用
- `leave_requests` - 空集合，未使用

## 🔄 数据流转关系

### 家属端数据流
```
1. 登录流程
   wechat-login → employee（保存/更新用户信息）

2. 绑定流程
   bind-senior → family_members（建立家属-长者关联）

3. 数据查看
   home → elders（长者概况）
   care → daily_reports（护理日报）
   leave → leave_requests（请假记录）
   bill → bills（账单信息）
   所有页面 → branding（品牌配置）

4. 数据提交
   leave → leave_requests（新建请假申请）
```

### 管理端数据流
```
1. 数据统计
   admin-home → daily_reports、leave_requests、bills、elders（统计待审核数量）

2. 数据审核
   admin-daily → daily_reports（审核/编辑日报）
   admin-leave → leave_requests（审核/拒绝请假）
   admin-bill → bills（审核/生成账单）

3. 数据管理
   admin-elder → elders（增删改查老人信息）
   admin-data → elders、bills（批量导入）

4. 系统配置
   admin-brand → branding（更新品牌配置）
```

## 🔒 权限体系

### 角色定义
| 角色 | 类型 | 可访问页面 |
|------|------|-----------|
| 家属 | family | login、wechat-login、bind-senior、home、care、leave、bill |
| 护工 | nurse | 所有管理端页面 |
| 文员 | staff | 所有管理端页面 |
| 管理员 | admin | 所有管理端页面 |

### 权限验证方式
```javascript
// 家属端权限检查
const user = props.$w.auth.currentUser;
if (user?.type && user.type !== 'family') {
  // 无权限，重定向到登录页
  props.$w.utils.redirectTo({ pageId: 'login', params: {} });
}

// 管理端权限检查
const user = props.$w.auth.currentUser;
const allowedTypes = ['nurse', 'staff', 'admin'];
if (user?.type && !allowedTypes.includes(user.type)) {
  // 无权限，重定向到登录页
  props.$w.utils.redirectTo({ pageId: 'login', params: {} });
}
```

## 📝 开发规范

### 文件命名
- 页面文件：`/pages/{pageId}.jsx`
- 组件文件：`/components/{ComponentName}.jsx`
- 数据模型：使用小写加下划线命名，如 `daily_reports`

### 代码规范
- 使用 2 空格缩进
- 组件文件只导出一个默认组件
- 导出的组件放在文件末尾
- Props 类型为 IProps，但使用 jsx 语法

### 数据访问规范
```javascript
// 推荐方式：使用 callDataSource
const result = await props.$w.cloud.callDataSource({
  dataSourceName: 'collection_name',
  methodName: 'get',
  params: {}
});

// 复杂查询：使用云开发实例
const tcb = await props.$w.cloud.getCloudInstance();
const db = tcb.database();
const result = await db.collection('collection_name').where({...}).get();
```
